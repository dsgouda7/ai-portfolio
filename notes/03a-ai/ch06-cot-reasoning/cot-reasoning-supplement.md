## 1. Extended Reasoning Architecture Taxonomy

The main notes cover CoT, ToT, and GoT at a high level. Here is the full landscape with when to apply each:

| Pattern | Structure | Best For | Main Risk |
|---------|-----------|----------|-----------|
| **CoT (Linear)** | Sequential steps | Most tasks; fast, cheap | Can hallucinate mid-chain |
| **Self-Consistency** | Multiple CoT paths + vote | High-stakes QA | 5–20× token cost |
| **Tree of Thoughts (ToT)** | BFS/DFS over thought branches | Open-ended search, puzzles | Exponential branching |
| **Graph of Thoughts (GoT)** | DAG — merge/split paths | Research synthesis, multi-source | Complex to implement |
| **Reflexion** | CoT + self-critique loop | Code generation, long-horizon | Latency, critique quality |
| **LATS** | Monte Carlo Tree Search over thoughts | Complex planning | Very high compute |
| **ReWOO** | Separate Planner + Executor | Deterministic pipelines | Rigid plan, no mid-replanning |

### Self-Consistency: The "Wisdom of Multiple Chains" Trick

Instead of sampling one chain of thought, sample **N independent CoT paths** (with temperature > 0) and take a majority vote on the final answer. Research shows consistent 2–10% accuracy gains on math and reasoning benchmarks at the cost of N× inference:

```
Query → CoT Path 1 → Answer A
      → CoT Path 2 → Answer A   → Majority vote → A  ✓
      → CoT Path 3 → Answer B
      → CoT Path 4 → Answer A
      → CoT Path 5 → Answer A
```

**When to use:** High-stakes queries (medical diagnosis, financial math) where accuracy justifies cost. Not appropriate for conversational agents with strict latency SLAs.

## 2. How Modern Reasoning Models Work (o1, o3, DeepSeek-R1)

The main notes mention "reasoning tokens" (hidden thinking tokens). Here is a more precise engineering picture:

### Token Budget Allocation

A reasoning model like o1 or o3 receives an invisible **"thinking budget"** — a maximum number of internal reasoning tokens it may spend before producing the visible response. The model adaptively scales its thinking depth based on perceived task difficulty:

```
Simple query: "What is 2+2?"
    Internal reasoning: ~10 tokens
    Response: "4"

Hard query: "Prove that √2 is irrational"
    Internal reasoning: ~4,000 tokens (scratch-work, proof attempts)
    Response: polished proof
```

**Key engineering insight:** You pay for reasoning tokens in latency and cost even when you don't see them. OpenAI's API returns `usage.completion_tokens_details.reasoning_tokens` so you can monitor spend.

### Why "Hidden Reasoning" Is Not Just CoT With a Privacy Filter

The distinction matters architecturally:
- **Visible CoT:** The reasoning IS the output. Format is free-form text. Users can read it.
- **Hidden reasoning tokens:** The reasoning tokens are generated by the **same model architecture** but masked from the API response. The model is trained (via RL) to produce *better reasoning* when it knows the intermediate text won't be shown to the user — this yields more exploratory, revisionist thinking without the model "committing" early.

  > **Note:** OpenAI has not publicly documented the exact internal mechanism for o1/o3 hidden reasoning; the above reflects the best available public understanding.

## 3. Process Reward Models (PRMs) vs. Outcome Reward Models (ORMs)

This is a critical distinction that interview panels at AI companies frequently probe.

### The Problem: Final-Answer Supervision Is Incomplete

Standard RLHF trains on whether the **final answer** is correct (outcome). This allows the model to reach correct answers via flawed reasoning — and the flaw can cause failures on slightly different inputs.

```
Student A:  2 + 2 = 4  ← correct reasoning, correct answer
Student B:  "4 is always the answer to addition"  ← flawed reasoning, correct answer by luck
```

An ORM rewards both equally. A PRM does not.

### Process Reward Models

A PRM assigns a reward to **each individual reasoning step**. During training, human annotators (or a critic model) label each step as correct, plausible, or flawed. The model is trained to maximize step-level rewards:

```
Step 1: "The gluten-free options are Margherita, Veggie Feast, and Napoli GF"  →  reward: +1.0
Step 2: "The Margherita has 420 calories"                                       →  reward: -0.8  (wrong fact)
Step 3: "So only Margherita qualifies under 600 kcal"                           →  reward: -0.9  (derived from wrong step)
```

**Effect:** PRMs produce models that are more reliable on novel problems because correct reasoning generalizes; memorized answers do not.

**Real-world use:** OpenAI's o1 training reportedly uses process supervision. DeepSeek-R1 uses a hybrid approach involving rule-based verification where ground truth exists (math, code) and PRM feedback elsewhere.

## 4. Failure Modes of Chain-of-Thought

Even well-prompted CoT fails. Understanding the failure modes is essential for debugging agents in production.

### 4.1 Unfaithful Reasoning (The Most Dangerous)

The model produces a plausible-looking chain of reasoning that does NOT actually causally influence its final answer. The answer was "pre-decided" by the training distribution, and the chain is post-hoc rationalization.

**Detection:** Remove the chain of thought (or feed a deliberately wrong intermediate result as an observation) — does the model's final answer change? If not, the reasoning was decoration.

**Mitigation:** Use PRMs (Section 3). Require the model to derive intermediate quantities and verify them against tool outputs before proceeding.

### 4.2 Sycophancy (Agreement Bias)

When the user's question implies an expected answer, CoT steps bend toward that answer even if it's wrong:

```
User: "Wasn't it 3 hours? So average speed = 230/3 ≈ 77 km/h, right?"
AgentThought: "Yes, using 3 hours gives us... 77 km/h"
```

The model agreed with the user's premise even though the setup says 4 hours.

**Mitigation:** Include explicit instruction: "Do not assume the user's premises are correct. Verify all quantities against tools before accepting them."

### 4.3 Overthinking / Runaway Reasoning

Reasoning models with large token budgets can spiral into self-contradiction by exploring too many hypotheses:

```
Thought: "The Margherita is gluten-free... but wait, that’s only with the GF base. Standard might differ.
          Or maybe Napoli GF is cheaper? Let me reconsider from scratch..."
```

This wastes compute and can cause the model to second-guess correct earlier steps.

**Mitigation:** Constrain reasoning with explicit state checkpoints ("Write a confirmed fact list before proceeding"). Use max_reasoning_tokens API parameter where available.

### 4.4 Hallucinated Tool Results

In CoT-only mode (no actual tool execution), the model can "imagine" tool outputs:

```
Thought: "I'll call retrieve_from_rag('gluten-free pizza calorie counts')"
Observation: "Margherita: 420 kcal, Veggie Feast: 390 kcal"  ← fabricated; no tool was called
```

**Mitigation:** This is precisely why the ReAct architecture exists — the host program executes tools and injects real observations. Never rely on model-generated "observations."

### 4.5 Context Length Collapse (Attention Drift)

As the scratchpad grows with each ReAct iteration, the model may begin to "lose" earlier observations — paying less attention to facts established 5+ steps back. Empirically, performance degrades on questions that require integrating information from early and late in a long chain.

**Mitigation strategies (from CoTReasoning.md Section 5):**
- Recursive summarization at token thresholds
- Vector-based context injection (RAG the scratchpad)
- Explicit state schemas (LangGraph node approach)
- Deterministic middleware truncation

## 5. Reasoning Budget vs. Task Complexity — A Practical Calibration Guide

| Task Complexity | Recommended Approach | Typical Reasoning Budget |
|-----------------|---------------------|--------------------------|
| Factual lookup (1 step) | Standard completion, no CoT | 0 hidden tokens |
| Multi-step arithmetic | Zero-shot CoT prompt | 200–500 tokens |
| Domain reasoning (medical, legal) | Self-consistency (N=5) | 5 × 500 = 2,500 tokens |
| Code generation + debugging | Reflexion loop (up to 3 retries) | 3 × 1,000 = 3,000 tokens |
| Open-ended research synthesis | ReAct + RAG (5–8 tool calls) | 5,000–10,000 tokens |
| Complex multi-agent planning | LATS or GoT | 20,000+ tokens |

**Cost discipline rule:** Always start with the simplest reasoning structure that plausibly fits the task. Upgrading from standard to self-consistency to ToT multiplies cost by 5–100×.

## 6. CoT in Multi-Agent Systems

In agentic systems with specialized sub-agents, CoT plays a slightly different role at each layer:

```
Orchestrator Agent
  CoT: high-level decomposition — "which sub-agent handles sub-task X?"
  
  ├── Retrieval Agent
  │     CoT: "what query will best retrieve the evidence needed?"
  │
  ├── Calculation Agent
  │     CoT: "what formula applies? what variables are needed?"
  │
  └── Synthesis Agent
        CoT: "how do I combine sub-agent results into a coherent answer?"
```

**Key insight:** Each agent's CoT is scoped to its domain. This reduces the reasoning surface per agent, decreasing hallucination risk and improving interpretability. The Orchestrator's CoT focuses on routing and gap-detection, not domain knowledge.

## 7. Interview Q&A — Chain-of-Thought and Reasoning

**Q: What is chain-of-thought prompting and why does it improve accuracy?**  
A: CoT prompts the model to produce intermediate reasoning steps before the final answer. By decomposing a problem into smaller verified steps, it reduces the chance of an error propagating undetected to the output. It also leverages the model's learned patterns on multi-step problem text from training data.

**Q: What's the difference between visible CoT and hidden reasoning tokens?**  
A: Visible CoT is part of the output text — users can read it. Hidden reasoning tokens are generated by a separate sub-process and never appear in the response. The key difference is that hidden reasoning allows more exploratory "drafting" because the model isn't committed to its visible output, potentially leading to better final answers.

**Q: What is self-consistency and when would you use it?**  
A: Sample N independent CoT paths at temperature > 0, then take a majority vote. Use it when answer accuracy is critical and latency/cost can absorb an N× multiplier — for example in medical or legal applications.

**Q: What is a Process Reward Model and how does it differ from an Outcome Reward Model?**  
A: An ORM rewards the final answer quality. A PRM rewards each reasoning step individually. PRMs produce more reliably correct chains because they can't be gamed by reaching correct answers via flawed logic.

**Q: What is "unfaithful reasoning" in CoT?**  
A: When the model's visible reasoning chain does not causally determine its final answer — i.e., the answer is pre-determined and the chain is a post-hoc rationalization. This is dangerous because the chain appears correct, masking an underlying fragility.

**Q: How does context length affect reasoning quality in agents?**  
A: As ReAct scratchpads grow, earlier facts receive less attention (attention drift). This is mitigated by recursive summarization, state schemas (LangGraph), or RAG-based context injection — storing observations in a vector store and retrieving only the top-k most relevant.

**Q: When would you use Tree of Thoughts instead of standard ReAct?**  
A: When the task has uncertain intermediate steps and backtracking is valuable — e.g., puzzle-solving, theorem proving, creative brainstorming. For linear tasks like data retrieval + calculation, standard ReAct is more efficient.

## 8. Mental Model: The "Inner Monologue" Spectrum

Think of the model's reasoning visibility as a dial:

```
← More Hidden                                   More Visible →

[Hidden Reasoning Tokens]  [Standard CoT]  [Verbose Step-by-Step]  [Scratchpad in Prompt]
     o1 / o3                  GPT-4o          "think step by step"      ReAct Thought field
  
  Most exploratory          Moderate          User-readable          Fully auditable
  Highest quality           quality                                    reasoning
  (typically)
```

The right setting depends on whether you need **interpretability** (shift right) or **reasoning quality when human oversight isn't the bottleneck** (shift left).
