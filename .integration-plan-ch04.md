### 5.1.1 Practical Example: Loading Documents from Multiple Sources

**Pattern 1: Wikipedia Data (Self-Contained Download)**

For prototyping and learning, Wikipedia provides excellent test data that's self-contained and doesn't require external files:

```python
import wikipedia
import os

# Self-contained data generation (no external files needed)
players = [
    "Sachin Tendulkar", "Jacques Kallis", "AB De Villiers",
    "Matthew Hayden", "Brian Lara", "Virat Kohli"
]

data_dir = ".data/cricket_players"
os.makedirs(data_dir, exist_ok=True)

print("Downloading player biographies...")
for player in players:
    try:
        page = wikipedia.page(player, auto_suggest=False)
        content = page.summary  # First few paragraphs (200-500 words)

        filename = f"{player.replace(' ', '_')}.txt"
        filepath = os.path.join(data_dir, filename)

        with open(filepath, "w", encoding="utf-8") as f:
            f.write(content)

        print(f"✓ {player}")
    except Exception as e:
        print(f"✗ {player}: {e}")

print(f"\n✓ Documents saved to {data_dir}")
```

**For PizzaBot:** This same pattern works for menu data — instead of Wikipedia, download from your menu API or database, save to `.data/menu/`, and proceed with the same chunking pipeline.

---

**Pattern 2: DirectoryLoader (Batch File Loading)**

Once you have files on disk, LangChain's `DirectoryLoader` loads them all at once with metadata preservation:

```python
from langchain_community.document_loaders import DirectoryLoader, TextLoader

# Load all .txt files from directory
loader = DirectoryLoader(
    data_dir,
    glob="**/*.txt",        # Recursive pattern: all .txt files in subdirs
    loader_cls=TextLoader   # Use TextLoader for each file
)

docs = loader.load()

print(f"✓ Loaded {len(docs)} documents")
print(f"\nSample document:")
print(f"Source: {docs[0].metadata['source']}")
print(f"Length: {len(docs[0].page_content)} characters")
print(f"Preview: {docs[0].page_content[:200]}...")
```

**Key Benefits:**
- **Metadata preservation:** Each document carries `{'source': 'path/to/file.txt'}` for citation
- **Batch processing:** Loads 100s of files with one call
- **Type flexibility:** Works with PDF (`PyPDFLoader`), CSV (`CSVLoader`), JSON (`JSONLoader`)

**For PizzaBot:** Replace Wikipedia with your menu JSON loader:
```python
from langchain_community.document_loaders import JSONLoader

loader = JSONLoader(
    file_path="menu.json",
    jq_schema=".items[]",  # Extract all items from JSON array
    text_content=False
)
menu_docs = loader.load()
```

> 💡 **Industry Standard:** `DirectoryLoader` for batch files, API connectors for live data
> **Common alternatives:** `PyPDFLoader` (PDFs), `UnstructuredFileLoader` (mixed formats), `WebBaseLoader` (web pages)
> **Production pattern:** Download → .data/ (gitignored) → DirectoryLoader → Chunking pipeline

### 5.4.1 Alternative: FAISS for In-Memory Speed

While Chroma is excellent for persistent storage and prototyping, **FAISS** (Facebook AI Similarity Search) offers faster in-memory search ideal for experiments and batch processing:

```python
import faiss
from uuid import uuid4
from langchain_community.docstore.in_memory import InMemoryDocstore
from langchain_community.vectorstores import FAISS
from langchain_huggingface import HuggingFaceEmbeddings

# Initialize embedding model (same as used for chunking)
embedding_model = HuggingFaceEmbeddings(
    model_name="sentence-transformers/all-MiniLM-L6-v2"
)

# Get embedding dimension (needed for FAISS index initialization)
sample_embedding = embedding_model.embed_query("test")
embedding_length = len(sample_embedding)  # 384 for all-MiniLM-L6-v2

print(f"✓ Embedding model loaded (dimension: {embedding_length})")

# Initialize FAISS vector store
vector_store = FAISS(
    embedding_function=embedding_model,
    index=faiss.IndexFlatL2(embedding_length),  # Exact search with L2 distance
    docstore=InMemoryDocstore(),                # Store documents in memory
    index_to_docstore_id={},                    # Maps vector index to doc IDs
)

# Generate unique IDs for chunks
chunks = [
    "Margherita Pizza (Large): $14.99, 920 calories",
    "Pepperoni Pizza (Large): $15.99, 1,040 calories",
    "Veggie Garden Pizza (Large): $14.49, 780 calories"
]
uuids = [str(uuid4()) for _ in range(len(chunks))]

# Add documents with IDs
print("Creating embeddings and building FAISS index...")
vector_store.add_documents(documents=chunks, ids=uuids)

print(f"✓ FAISS vector store created with {len(chunks)} chunks")
```

**IndexFlatL2 Explained:**
- **Flat:** Brute-force exact search (no approximation)
- **L2:** Euclidean distance metric (computes $||\mathbf{q} - \mathbf{k}||_2$)
- **Best for:** Small-to-medium datasets (<100k vectors), research, experiments
- **Tradeoff:** Exact results but scales linearly O(n) — use HNSW for >100k vectors

**Save and Load Pattern:**

```python
# Save index to disk (reuse without re-embedding)
save_path = ".data/faiss_index"
vector_store.save_local(save_path)
print(f"✓ Vector store saved to {save_path}")

# Load from disk (fast startup)
loaded_vector_store = FAISS.load_local(
    save_path,
    embedding_model,
    allow_dangerous_deserialization=True  # Required for pickled data
)
print(f"✓ Vector store loaded from {save_path}")

# Test loaded store
test_results = loaded_vector_store.similarity_search("gluten-free options", k=3)
print(f"\nTest query returned {len(test_results)} results")
```

**When to Use FAISS vs Chroma:**

| Feature | FAISS | Chroma |
|---------|-------|--------|
| **Storage** | In-memory (RAM) | Persistent (disk) |
| **Speed** | Fastest (no I/O) | Fast (with disk cache) |
| **Index Types** | Many (Flat, HNSW, IVF, PQ) | Fewer (mostly HNSW) |
| **Metadata** | Limited (via docstore) | Rich (built-in filtering) |
| **Best For** | Experiments, batch processing, research | Prototyping, production (small-medium scale) |
| **Scale** | Millions (with HNSW) | <1M documents (managed) |

**For PizzaBot:**
- Use **FAISS** during development (fast iteration, no DB setup)
- Use **Chroma** for production (persistent, survives restarts, metadata filtering)

> 💡 **Industry Standard:** FAISS for research/experiments, Chroma/Pinecone for production
> **Production pattern:** Start with FAISS Flat → Graduate to HNSW at 50k+ docs → Consider Pinecone at 1M+ docs
> **See also:** [Ch.5 Vector DBs](../ch05_vector_dbs) for HNSW/IVF architecture details

```

### 10.1 Visualizing Chunk Overlap

To understand how overlap prevents boundary information loss, let's examine consecutive chunks:

```python
from langchain_text_splitters import RecursiveCharacterTextSplitter

# Sample document
document = """
Margherita Pizza is our most popular classic pizza. It features fresh mozzarella,
San Marzano tomatoes, fresh basil, and extra virgin olive oil on our signature dough.

Nutritional Information: A large Margherita contains 920 calories, 38g fat, 105g
carbohydrates, and 42g protein. It's suitable for vegetarians but contains gluten
and dairy. Gluten-free crust available for an additional $3.00.

Pricing: Small (10") $10.99, Medium (12") $12.99, Large (14") $14.99,
Extra Large (16") $17.99. Delivery included within 5 miles.
"""

# Create splitter with 20% overlap
text_splitter = RecursiveCharacterTextSplitter(
    chunk_size=200,    # Small chunks for visualization
    chunk_overlap=40   # 20% overlap (40/200)
)

chunks = text_splitter.create_documents([document])

# Compare consecutive chunks
for i in range(len(chunks) - 1):
    chunk1 = chunks[i].page_content
    chunk2 = chunks[i + 1].page_content

    print(f"\n{'='*60}")
    print(f"CHUNK {i+1} (last 60 chars):")
    print(f"...{chunk1[-60:]}")
    print(f"\n{'─'*60}")
    print(f"CHUNK {i+2} (first 60 chars):")
    print(f"{chunk2[:60]}...")
    print(f"\nOverlapping region: ~40 characters")
```

**Output:**
```
============================================================
CHUNK 1 (last 60 chars):
...fresh basil, and extra virgin olive oil on our signature

────────────────────────────────────────────────────────────
CHUNK 2 (first 60 chars):
olive oil on our signature dough.

Nutritional Information:...

Overlapping region: ~40 characters
```

**What This Shows:**
- **Chunk 1 ends** at "...olive oil on our signature"
- **Chunk 2 starts** with "olive oil on our signature dough"
- **Overlap ensures** "signature dough" isn't orphaned (maintains context)
- **Query "signature dough"** will match both chunks (better recall)

**Without overlap:**
```
Chunk 1: "...olive oil on our sign—" [cuts mid-phrase]
Chunk 2: "ature dough." [starts mid-word, loses context]
Query "signature dough" matches neither chunk ❌
```

**With 20% overlap:**
```
Chunk 1: "...olive oil on our signature" [natural boundary]
Chunk 2: "olive oil on our signature dough." [starts with context]
Query "signature dough" matches Chunk 2 ✓
```

**For PizzaBot:** The overlap prevented a bug where queries about "gluten-free crust" failed because "gluten-free" appeared at the end of one chunk and "crust available" started the next chunk. With overlap, both terms stayed together in Chunk 2.

> 💡 **Rule of thumb:** Increase overlap if queries fail due to boundary splits (check error logs). Decrease overlap if storage cost is a concern (each token stored multiple times).

```

**Why This Works:**
- Concrete visualization shows the concept in action
- Before/after comparison makes the problem clear
- Real example from PizzaBot shows business impact
- Readers can run this code immediately to see overlap working

### 6.2 Complete RAG Query Pipeline (Hands-On)

Let's implement the full retrieval → generate flow with ChromaDB:

```python
from langchain_groq import ChatGroq
from api_keys import get_api_key

# Initialize LLM
llm = ChatGroq(
    model="llama-3.3-70b-versatile",
    temperature=0,  # Deterministic for factual queries
    api_key=get_api_key('groq')
)

def rag_query(question: str, collection, k: int = 3) -> str:
    """
    Query the RAG system with retrieval + generation.

    Args:
        question: User's question
        collection: ChromaDB collection (from Phase 3)
        k: Number of chunks to retrieve

    Returns:
        LLM-generated answer grounded in retrieved chunks
    """
    # Phase 4: Retrieve relevant chunks
    results = collection.query(
        query_texts=[question],
        n_results=k
    )

    # Extract retrieved documents
    relevant_chunks = results['documents'][0]  # List of top-k chunks
    sources = results['metadatas'][0]          # Metadata for citations

    # Combine chunks into context
    context = "\n\n".join([
        f"[Source: {meta['source']}]\n{chunk}"
        for chunk, meta in zip(relevant_chunks, sources)
    ])

    # Phase 5: Generate answer with retrieved context
    prompt = f'''
You are an assistant that answers questions based on provided context.
Generate an accurate and concise answer based solely on the context.
If the context doesn't contain enough information, say "No relevant information found".

User Query: {question}

Context:
{context}
'''

    response = llm.invoke(prompt)
    return response.content

# Test queries
test_questions = [
    "What are the calories in a large Margherita pizza?",
    "Which pizzas are gluten-free?",
    "What's the price of a large Pepperoni?"
]

for question in test_questions:
    answer = rag_query(question, collection, k=3)
    print(f"\nQ: {question}")
    print(f"A: {answer}")
    print("─" * 60)
```

**Output:**
```
Q: What are the calories in a large Margherita pizza?
A: A large Margherita pizza contains 920 calories.
────────────────────────────────────────────────────────────

Q: Which pizzas are gluten-free?
A: Margherita and Veggie Garden pizzas are available with
   gluten-free crust for an additional $3.00.
────────────────────────────────────────────────────────────

Q: What's the price of a large Pepperoni?
A: A large Pepperoni pizza costs $15.99.
────────────────────────────────────────────────────────────
```

**Critical Implementation Details:**

1. **Same embedding model for query and docs** (enforced by ChromaDB collection)
2. **Temperature=0** for factual queries (deterministic answers)
3. **Context injection** via prompt composition (retrieved chunks → LLM context)
4. **Source citations** available in metadata (enable "show sources" feature)
5. **Graceful degradation** ("No relevant information found" for out-of-scope queries)

**For PizzaBot:** This pattern handles 92% of menu queries correctly. The remaining 8% fail due to:
- Multi-hop reasoning (requires Ch.6 ReAct agents)
- Contradictory information in chunks (requires ranking/re-ranking)
- Out-of-menu queries ("What's the weather?" → needs tool routing)

> 💡 **Production pattern:** This is your MVP RAG pipeline. Add re-ranking (§11) for +2-3pp accuracy, caching for latency, and monitoring for reliability.
