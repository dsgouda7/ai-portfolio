The Oracle's

K-Nearest Neighbours

"You are who you stand next to." — The Oracle

algorithm · distance metrics · the K dial · curse of dimensionality
CH.1 — KNN
KNOCK KNOCK
👁️

THE ORACLE

You have a new point. No label. You want to know what it is.
Look around it. Find its K nearest neighbours. Vote. Take the majority.
That is all there is.

KAI

How do you measure "nearest"?

THE ORACLE

Euclidean distance for most cases. Manhattan when features have different scales. Cosine when direction matters more than magnitude.
CH.1 — KNN
from sklearn.neighbors import KNeighborsClassifier
from sklearn.preprocessing import StandardScaler
X_scaled = StandardScaler().fit_transform(X_train)
knn = KNeighborsClassifier(n_neighbors=5, metric='euclidean')
knn.fit(X_scaled, y_train)

How KNN Works

graph LR A[New point X] --> B[Compute distance to all training points] B --> C[Sort by distance] C --> D[Take K nearest] D --> E{Task?} E -->|Classification| F[Majority vote] E -->|Regression| G[Average value]

⚙️ The K Dial — Your First Bias-Variance Tradeoff

K is the lever that controls how much the model trusts its local neighbourhood.

K=1 (too small)
Memorises every training point. Perfect training accuracy. Falls apart on new data. High variance.
K=N (too large)
Averages everything. Always predicts the majority class regardless of position. High bias.
Sweet spot: Use cross-validation. Try K = 1, 3, 5, 7, 11, 21. Pick the K that minimises validation error.
🔒 Intuition Lock: K is the first dial you'll encounter in ML. Every algorithm has one. Tree depth (Ch.2), n_estimators (Ch.3), C (Ch.4) — all the same tradeoff. Small = overfit, large = underfit. Remember this.

The Curse of Dimensionality

When you have many features, all points become approximately equidistant. "Nearest" stops meaning anything.

Rule of thumb: KNN degrades badly beyond ~20 features without dimensionality reduction first (see Ch.6).

KAI

So K is a lever. High K = smooth but blind. Low K = sharp but fragile. I need to calibrate it.

THE ORACLE

Now you're thinking like a detective. Cross-validate. The data tells you the answer.

Decision Trees

"A tree that grows without restraint explains everything and predicts nothing." — Elder Gini

Gini impurity · information gain · depth · pruning · the same dial, different name
CH.2 — DECISION TREES
THE GROVE
🌳

ELDER GINI

Detective. K's voting worked — but it can't explain why it voted that way. My trees can.
At each node, I ask a question that most separates the classes. I measure separation with Gini impurity.

KAI

You keep splitting until it's pure?

ELDER GINI

Until it's pure — or until I stop it. That's the depth limit. Your second dial, detective. Same tradeoff as K.
🔒 Callback — The Dial Returns: K's tradeoff (Ch.1) and tree depth are the same thing wearing different clothes. Small depth = high bias, misses patterns. Unlimited depth = memorises noise. The optimal is always somewhere in between.
CH.2 — DECISION TREES

Gini Impurity — The Splitting Criterion

At each candidate split, Elder Gini computes how "mixed" each resulting group is:

$$\text{Gini}(S) = 1 - \sum_{i=1}^{C} p_i^2$$

A node is pure (Gini = 0) when all samples belong to one class. Mixed 50/50 gives Gini = 0.5 (worst).

📐 Worked Gini Calculation

Four data points: A🟢, B🟢, C🔴, D🔴. Two candidate splits on feature "height > 5cm":

Split 1: Left={A,B,C}, Right={D}
Left: 2 green, 1 red → Gini = 1 − [(2/3)² + (1/3)²] = 1 − [0.444 + 0.111] = 0.444
Right: 0 green, 1 red → Gini = 1 − [0² + 1²] = 0.000
Weighted: (3/4)×0.444 + (1/4)×0.000 = 0.333
Split 2: Left={A,B}, Right={C,D}
Left: 2 green, 0 red → Gini = 1 − [1² + 0²] = 0.000
Right: 0 green, 2 red → Gini = 1 − [0² + 1²] = 0.000
Weighted: (2/4)×0.000 + (2/4)×0.000 = 0.000
Winner: Split 2. Perfect purity. The tree uses this question.

The Tree-Building Loop

graph TD A[All training data at root] --> B[Try every possible split on every feature] B --> C[Pick split with lowest weighted Gini] C --> D{Pure enough or max depth?} D -->|No| E[Recurse on each child] D -->|Yes| F[Leaf — assign majority class] E --> B 
from sklearn.tree import DecisionTreeClassifier, export_text
tree = DecisionTreeClassifier(max_depth=4, criterion='gini')
tree.fit(X_train, y_train)
print(export_text(tree, feature_names=feature_names))

Strengths and Weaknesses

✅ Interpretable — you can draw it
✅ Handles mixed feature types
✅ No feature scaling needed
✅ Naturally handles multi-class
❌ Overfits aggressively without pruning
❌ Unstable — small data changes = different tree
❌ Axis-aligned splits only
❌ Biased toward high-cardinality features

Random Forests & Gradient Boosting

Two solutions to Elder Gini's instability problem.
One builds a democracy. The other builds a correction chain.

bagging · feature randomisation · boosting · residuals · learning rate · n_estimators
CH.3 — ENSEMBLES
THE BARRACKS
📦

GEN. BAGGING

Elder Gini's trees are brittle. Mine aren't. I bootstrap — sample the data with replacement — and grow each tree on a different random subset.
Then I also randomise which features each tree may consider at each split. Independent errors cancel out.
🎯

GEN. BOOSTING

My approach is different. I build trees sequentially. Each tree looks at what the previous ones got wrong — the residuals — and specialises in fixing those mistakes.
Learning rate controls how much each tree contributes. Slow learning, many trees = better generalisation.

KAI

n_estimators — that's the dial again. More trees, more time, but diminishing returns after some point.
🔒 Callback — Third Manifestation: K in Ch.1. Depth in Ch.2. n_estimators here. Every algorithm has this control. Low = underfit, high = overfit or just wasteful. Same dial. Different label.
graph TD subgraph Bagging D1[Data] --> B1[Bootstrap 1] & B2[Bootstrap 2] & B3[Bootstrap N] B1 --> T1[Tree 1] B2 --> T2[Tree 2] B3 --> TN[Tree N] T1 & T2 & TN --> V[Majority Vote] end
CH.3 — ENSEMBLES

Boosting: The Residual Chain

graph LR A[Data] --> T1[Tree 1: predict Y] T1 --> R1[Residuals = Y - pred1] R1 --> T2[Tree 2: predict residuals] T2 --> R2[New residuals] R2 --> T3[Tree 3] T3 --> F[Final = sum of all trees x lr]

Random Forest — Key Hyperparams

  • n_estimators — number of trees (100–500 typical)
  • max_features — features per split (√p for classification)
  • max_depth — tree depth limit (None is common)
  • min_samples_leaf — prevents tiny leaves
Out-of-bag (OOB) samples give free validation — no separate val set needed.
from sklearn.ensemble import RandomForestClassifier
rf = RandomForestClassifier(n_estimators=200,
    max_features='sqrt', oob_score=True)
rf.fit(X_train, y_train)
print(rf.feature_importances_)

Gradient Boosting — Key Hyperparams

  • n_estimators — number of boosting rounds
  • learning_rate — shrinkage (0.01–0.3)
  • max_depth — shallow trees (3–6) work best
  • subsample — row sampling per round (stochastic GB)
High learning_rate with many trees = overfitting. Lower lr + more trees almost always wins.
Feature Importance: Both models provide feature importance scores — how often / how much each feature contributed across all trees. Free feature selection signal.

Chapter 4

The Knight
of Margins

Support Vector Machines

"I do not care about all the points. I care only about the ones that define the boundary." — The Knight

hyperplane · margin · support vectors · C parameter · kernel trick · RBF
CH.4 — SVM
THE BOUNDARY
⚔️

THE KNIGHT

The forests trust numbers. I trust geometry. My goal is a hyperplane that separates classes with the maximum possible margin.
Wider margin = more confident separation = better generalisation. The points that define the margin are the support vectors. Everything else is irrelevant.

The Optimisation Problem

$$\min_{w,b} \frac{1}{2}\|w\|^2 + C\sum_i \xi_i$$ $$\text{subject to: } y_i(w \cdot x_i + b) \geq 1 - \xi_i$$

$\xi_i$ are slack variables — they allow some misclassification. C controls the penalty.

KAI

C is the dial.

THE KNIGHT

Yes. High C — I tolerate no misclassification. Narrow margin, risk of overfit. Low C — I allow errors to keep the margin wide. Smoother boundary.
🔒 Callback — Fourth Dial: Ch.1: K. Ch.2: depth. Ch.3: n_estimators. Now C. Every algorithm has a regularisation knob. The physics is always the same: tighter fit to training = worse generalisation.
CH.4 — SVM

The Kernel Trick — Escaping Linearity

When classes are not linearly separable, the kernel maps data to a higher-dimensional space where they become separable — without computing coordinates explicitly.

graph LR A[2D data — not separable] -->|Kernel φ| B[High-dim feature space] B --> C[Linear hyperplane found] C -->|Map back| D[Curved boundary in 2D]

Common Kernels

  • Linear: $K(x,z) = x \cdot z$ — fast, works well when features >> samples
  • RBF (Gaussian): $K(x,z) = e^{-\gamma\|x-z\|^2}$ — most versatile, two params (C, γ)
  • Polynomial: $K(x,z) = (x\cdot z + r)^d$ — good for image features

When to Use SVM

✅ High-dimensional data (text, genomics)
✅ Small-to-medium datasets
✅ Clear margin of separation exists
❌ Slow on >100k samples
❌ Sensitive to feature scale — always normalise
❌ No native probability output
Practical Note: Always standardise features before SVM. The margin depends on distances — unscaled features will dominate.

t-SNE & UMAP — Dimensionality Reduction

The data had 512 dimensions. Nobody could see it.
The Cartographer folded it into two.

CH.6 — t-SNE / UMAP
FOLDING SPACE
🗺️

CARTOGRAPHER

Detective. Your five classifiers are blind. They work in spaces they cannot see. I will show you what the data actually looks like.
High-dimensional data lives on a low-dimensional manifold. I find that manifold and project it to 2D — preserving local neighbourhood structure.

KAI

Is this for analysis or for building a model?

CARTOGRAPHER

Analysis only. Never feed t-SNE coordinates into a downstream model. The distances between clusters are meaningless. Only local structure is preserved.
Critical: t-SNE axes have no units. Cluster size and inter-cluster distance are artifacts of perplexity, not the data.

t-SNE vs UMAP

t-SNE
Preserves local structure extremely well. Cluster shapes meaningful. Slow on >10k points. Non-deterministic. Perplexity is the main knob (5–50). Random seed matters.
UMAP
Faster, scales to millions. Preserves both local AND some global structure. n_neighbors (local) + min_dist (compactness). More stable across runs. Preferred for most modern use.

The Workflow

graph LR A[High-dim data, e.g. 512-d] --> B[Reduce with PCA to 50-d] B --> C[Apply t-SNE or UMAP to 2-d] C --> D[Scatter plot — colour by label] D --> E{Clusters visible?} E -->|Yes| F[Data has exploitable structure — classifiers will work] E -->|No| G[Overlapping → harder problem, more features needed]
PCA first (to 50-d) dramatically speeds up t-SNE with minimal information loss.

⚙️ K-Means: What Came Before

Before Dr. Density arrives, Kai encountered K-Means — the simplest clustering algorithm. Know it, because the chapters ahead argue against it.

How K-Means Works

graph TD A[Choose K — number of clusters] --> B[Place K centroids randomly] B --> C[Assign each point to nearest centroid] C --> D[Recompute centroid as mean of assigned points] D --> E{Centroids changed?} E -->|Yes| C E -->|No| F[Done — K clusters found]
What it needs:
✅ Fast — O(n·K·iterations)
✅ Works well on globular, similar-sized clusters
✅ Scales to large datasets
Where it breaks:
❌ You must specify K upfront
❌ Non-spherical cluster shapes
❌ Clusters of very different density
❌ No concept of noise — every point gets assigned
❌ Sensitive to centroid initialisation 
from sklearn.cluster import KMeans

km.fit(X)
labels = km.labels_     # cluster per point
centers = km.cluster_centers_

Chapter 7

The Shapeless
Plague

DBSCAN & HDBSCAN — Density Clustering

K-Means needed to know the answer before it started.
Dr. Density asks only one question: is this region dense?

core points · border points · noise · eps · min_samples · HDBSCAN hierarchy
CH.7 — DBSCAN / HDBSCAN
DENSITY RISING
🦠

DR. DENSITY

Detective. The Cartographer showed you the map. But maps have shapes K-Means cannot follow. K-Means needs a K. K-Means needs round blobs. Real data has neither.
I find clusters of arbitrary shape. I reject noise as outliers. I need no prior knowledge of how many clusters exist.

KAI

Then what do you need?

DR. DENSITY

Two things. A neighbourhood radius — eps. And a minimum density — min_samples. That's all.
eps: How close two points must be to be "neighbours."
min_samples: How many neighbours a core point needs.

DBSCAN — Point Classification

🔵
Core Point

Has ≥ min_samples neighbours within eps. Backbone of a cluster.

🟡
Border Point

Within eps of a core point but not dense enough itself. Attached to cluster edge.

🔴
Noise Point

Not reachable from any core point. Labelled −1. This is DBSCAN's superpower — explicit outlier detection.

CH.7 — DBSCAN / HDBSCAN

DBSCAN Algorithm

graph TD A[Pick an unvisited point P] --> B{Neighbours within eps ≥ min_samples?} B -->|No| C[Mark as Noise — label=-1] B -->|Yes| D[Start new cluster — P is core point] D --> E[Add all density-reachable points to cluster] E --> F[Move to next unvisited point] C --> F F --> G{Any unvisited left?} G -->|Yes| A G -->|No| H[Done — clusters + noise identified]

HDBSCAN — The Upgrade

DR. DENSITY

DBSCAN has one critical weakness. eps is a global parameter. If your data has clusters of varying density — dense downtown, sparse suburbs — a single eps fails both regions simultaneously.
HDBSCAN removes eps entirely. It builds a hierarchy of clusters across all density levels, then extracts the most stable ones. Variable-density clusters, handled automatically.
HDBSCAN advantages:
✅ No eps to tune
✅ Handles variable density
✅ Soft cluster membership scores
✅ Better outlier scores
✅ Usually more stable results
Only param that matters:
min_cluster_size — minimum points to form a persistent cluster.
Start with 5% of dataset size.

DBSCAN vs HDBSCAN vs K-Means

from sklearn.cluster import DBSCAN
import hdbscan
# DBSCAN
db = DBSCAN(eps=0.5, min_samples=5)
labels = db.fit_predict(X)  # -1 = noise
# HDBSCAN
hdb = hdbscan.HDBSCAN(min_cluster_size=50)
labels = hdb.fit_predict(X)

Feature Engineering & Selection

"The model is only as smart as the features you give it. Bad features, brilliant algorithm — still bad results." — The Alchemist

CH.8 — FEATURE ENGINEERING
THE LAB
⚗️

THE ALCHEMIST

Detective. You're feeding raw ore to precision instruments. No wonder they struggle. Let me show you transformation.
Every feature must be examined. What is its range? Its distribution? Its relationship to the target? Raw data almost never speaks the algorithm's language.

KAI

K and SVM both needed scaled features. The ensemble methods didn't. So scaling isn't universal.

THE ALCHEMIST

Correct. Know your algorithm. Know what it assumes about the input space.

The Transformation Toolkit

graph LR A[Raw Feature] --> B{What type?} B -->|Numerical| C{Distribution?} B -->|Categorical| D{Cardinality?} B -->|DateTime| E[Extract: hour, day, month, is_weekend] C -->|Normal| F[StandardScaler: mean=0, std=1] C -->|Skewed| G[Log / BoxCox transform first] C -->|Unknown| H[RobustScaler if outliers present] D -->|Low: ≤10 categories| I[OneHotEncoder] D -->|High: >10 categories| J[Target / Ordinal encoding]

Feature Selection Methods

  • Variance threshold: Drop near-zero variance features
  • Correlation filter: Drop features correlated >0.95 with each other
  • Random Forest importance: Rank by mean decrease in impurity
  • Permutation importance: More reliable, model-agnostic
  • RFE: Recursive feature elimination
  • L1 (Lasso): Drives weak feature weights to zero

The Leakage Trap

Data leakage: Including information in your features that would not be available at prediction time. The model looks impressive in validation but collapses in production.
Leakage via scaling: Fitting StandardScaler on the full dataset before train/test split. The test set's statistics contaminate the scaler. Always fit on train, transform both.
🔒 Pipeline Rule: Every transformation that uses training data statistics (scaler, imputer, encoder) must be inside a sklearn Pipeline or fitted only on training data. If it sees the test set during fit, you have leakage. 
from sklearn.pipeline import Pipeline
from sklearn.preprocessing import StandardScaler
from sklearn.ensemble import GradientBoostingClassifier
pipe = Pipeline([
    ('scaler', StandardScaler()),
    ('clf', GradientBoostingClassifier(n_estimators=100))
])
pipe.fit(X_train, y_train)  # scaler sees only X_train

Kai had five classifiers. All claimed 95% accuracy. The Judges showed that accuracy, alone, is almost never enough.

CH.9 — EVALUATION
THE VERDICT

KAI

Five classifiers. All say 95% accurate on the fraud dataset. But 95 out of 100 transactions are legitimate anyway. What does 95% even mean?
If my model always predicts "not fraud," it gets 95% accuracy for free. And catches zero criminals.
The Accuracy Trap:
In imbalanced datasets, accuracy is misleading. A classifier that always predicts the majority class can achieve high accuracy while being completely useless.
This is why the Judges exist. Accuracy reports the total score. The Judges examine what kind of errors were made.

The Confusion Matrix

Predicted PositivePredicted Negative
Actually PositiveTPFN
Actually NegativeFPTN
$$\text{Precision} = \frac{TP}{TP+FP}$$ $$\text{Recall} = \frac{TP}{TP+FN}$$ $$F_1 = \frac{2 \cdot P \cdot R}{P + R}$$

Precision vs Recall Trade-off

High Precision, lower Recall: When you predict positive, you're usually right — but you miss many positives. Good when false alarms are costly (spam filter).
High Recall, lower Precision: You catch almost all positives — but generate false alarms. Better when missing a positive is catastrophic (cancer screening, fraud).

ROC-AUC

Plots True Positive Rate vs False Positive Rate at all thresholds. AUC = area under the curve.

$$AUC = 0.5 \text{ (random)} \quad AUC = 1.0 \text{ (perfect)}$$
AUC is threshold-agnostic. Use it when comparing models before choosing an operating threshold. Use F1 when the threshold and class balance are fixed.

Cross-Validation — Honest Evaluation

from sklearn.model_selection import StratifiedKFold, cross_val_score
from sklearn.metrics import f1_score, make_scorer
cv = StratifiedKFold(n_splits=5, shuffle=True, random_state=42)
scores = cross_val_score(model, X, y,
    cv=cv, scoring=make_scorer(f1_score))
print(f'F1: {scores.mean():.3f} ± {scores.std():.3f}')
Kai knows how to measure. One question remains: when the boundary cannot be described explicitly, what does the model learn from?

Activation Functions & Neural Network Fundamentals

Kai had mastered pattern detection without neural nets. But some problems required a different kind of architecture. He needed to understand what makes a network actually learn.

sigmoid · ReLU · tanh · Leaky ReLU · Softmax · vanishing gradients · backprop
CH.10 — ACTIVATIONS
⚡

GATE KEEPER

You built five classical models. All of them make one assumption: the boundary can be described explicitly — by distance, by split, by margin. What happens when no explicit description exists?
Neural networks learn representations. They don't need you to engineer features or describe the boundary. They discover it — layer by layer. But only if the activations are correctly chosen.

KAI

What's an activation function?

GATE KEEPER

The gate at each neuron. Without it, the entire network collapses to a single linear transformation — no matter how many layers you stack. The activation is what gives the network its power to model non-linearity.

Why Activations Exist — The Linear Collapse Problem

$$\text{No activation: } y = W_3(W_2(W_1 x)) = W_{combined} \cdot x \quad \text{(still linear)}$$ $$\text{With activation: } y = W_3\sigma(W_2\sigma(W_1 x)) \quad \text{(non-linear)}$$

Without a non-linear activation, stacking layers gives you nothing. The whole network is equivalent to a single matrix multiplication.

The Four Activations Kai Needs to Know

Sigmoid
$\sigma(x) = \frac{1}{1+e^{-x}}$
✅ Output in (0,1) — good for binary output layer
❌ Vanishing gradients for deep networks
❌ Not zero-centred
ReLU — Rectified Linear Unit
$f(x) = \max(0, x)$
✅ Default for hidden layers
✅ Sparse activation, computationally cheap
❌ Dying ReLU: if neuron always outputs 0, gradient = 0, weights freeze permanently
Leaky ReLU
$f(x) = \max(\alpha x, x), \; \alpha=0.01$
✅ Fixes dying ReLU — small negative slope
✅ Good for deep networks
Softmax
$\sigma(z)_j = \frac{e^{z_j}}{\sum_k e^{z_k}}$
✅ Multi-class output layer — converts logits to probabilities
❌ Sensitive to large logit values — use with proper initialisation
🔒 Vanishing Gradient: In deep networks with sigmoid/tanh, gradients shrink exponentially as they backpropagate through layers. Early layers learn almost nothing. ReLU largely solves this by having a constant gradient of 1 for positive inputs. Batch normalisation and residual connections solve it for very deep architectures.
tanh — Hyperbolic Tangent
$f(x) = \tanh(x) = \frac{e^x - e^{-x}}{e^x + e^{-x}}$
✅ Zero-centred — better gradient flow than sigmoid
✅ Output in (−1, 1) — useful for bounded representations
❌ Still saturates for large |x| — use ReLU for deep hidden layers

Common in RNNs and LSTM gates where bounded output matters.

Backpropagation — How Weights Actually Update

After every forward pass, the network computes a loss and pushes error backward through every layer via the chain rule:

$$\frac{\partial L}{\partial w} = \frac{\partial L}{\partial a} \cdot \frac{\partial a}{\partial z} \cdot \frac{\partial z}{\partial w}$$

The term $\frac{\partial a}{\partial z}$ is the activation's derivative. For sigmoid/tanh this approaches 0 at saturation — gradients vanish. For ReLU it is exactly 1 when $z > 0$ — gradients flow freely.

    nn.Linear(64, 32),
    nn.ReLU(),
    nn.Linear(32, 10),
    nn.Softmax(dim=1)   # output: Softmax for multi-class

The Codex — Principles Kai Earned (Ch.1–Ch.19):
  1. Every algorithm has a complexity dial. Tune it with cross-validation, not intuition.
  2. The bias-variance tradeoff is not a problem to solve — it is a constraint to navigate.
  3. Interpretability and accuracy are a spectrum, not a binary choice.
  4. High-dimensional data requires projection before visualisation.
  5. Real clusters have arbitrary shapes. K-Means is a starting point, not a law.
  6. Garbage features produce garbage predictions regardless of algorithm quality.
  7. Accuracy is almost always the wrong metric. Start with the confusion matrix.
  8. Data leakage is the most common cause of surprisingly good results.
  9. Non-linearity is the foundation of representation learning. Activations are not optional.
  10. A single line cannot separate XOR. One hidden layer can. Depth is how models compose abstraction.
  11. Backpropagation is just the chain rule, repeated. Optimisers are just momentum plus per-parameter scaling.
  12. Regularisation is a vow of less. L2 shrinks, L1 selects, Dropout randomises — all trade training fit for generalisation.
  13. Parameter sharing (CNN kernels) and sequential state (RNN cells) are two escapes from the curse of dimensionality.
  14. Attention is a soft dictionary lookup. Every modern model is stacks of it with residuals and LayerNorm around each layer.
  15. The model is not the answer. The model is the question crystallised into arithmetic.
