PROLOGUE
PROLOGUE

Four Layers That Make a Production AI System

Neural Chronicles stopped at the model — at transformers, attention, and the loss functions that train them. That is where applied AI begins, not where it ends. A production system has four layers stacked on top of the model, and an engineer who understands only one of them cannot ship. This codex covers all four.

LAYER 1 — THE MODEL & ITS SINGLE AGENT

A transformer becomes useful only after tokenisation, sampling, alignment (SFT + RLHF), and — for tasks bigger than one prompt — a reasoning loop that can call tools and retrieve evidence. Part I covers everything between "the model predicts the next token" and "the assistant finished the task I gave it."

Running example: PizzaBot — a single-agent ReAct loop that takes a pizza order end to end.

LAYER 2 — COORDINATION

One agent is simple. Many agents cooperating is a distributed system. You need wire formats (OpenAI messages), tool protocols (MCP), agent-to-agent protocols (A2A), durable event buses, shared memory, and a trust model. Skip this and "multi-agent" means a glorified for-loop that falls apart on any real load.

Running example: OrderFlow — a B2B purchase-order automation platform.

LAYER 3 — BEYOND TEXT

Images, video, and audio are not text. They use a different encoder stack (ViT, CLIP), a different generative family (diffusion), and a different sampling story (DDIM, DPM-Solver, classifier-free guidance). Part III treats the multimodal layer as a first-class peer of the text stack, not a bolt-on.

Running example: PixelSmith — a local creative studio that must run on a laptop GPU.

LAYER 4 — THE IRON

Everything above runs on GPUs. Part IV is the bill: VRAM budgets, arithmetic intensity, quantisation, parallelism, continuous batching, networking. This is where "why is my 7B model three times more expensive than it should be?" gets answered.

Running example: InferenceBase — a team deciding whether to self-host Llama-3-8B instead of paying $80k/month in API fees.

PART I · CH.1 LLM FUNDAMENTALS
AGENT

1A — Tokenisation & Sampling

A language model never sees text. It sees a sequence of integer IDs from a fixed vocabulary. The pipeline from string to model input, and from model output back to string, is where most "inexplicable" behaviour — off-by-one counts, currency formatting quirks, multilingual fragility — actually originates.

Byte-Pair Encoding (BPE) — how the vocabulary is built

  1. Start with a character-level vocabulary.
  2. Count all adjacent character pairs in the training corpus.
  3. Merge the most frequent pair into a new token. t+h → th.
  4. Repeat until the vocabulary hits the target size (32k–100k).

Result: common words become one token (the, model); rare words split (trans+former); code is often single-character dense; numbers tokenise byte-by-byte — which is the reason arithmetic is hard.

Rule of thumb: 1 token ≈ 0.75 English words ≈ 4 bytes. A million tokens ≈ 4 MB of text. The same text tokenises differently across models — never assume a GPT-4 count matches Claude.

The context window

Model classWindow (tokens)
GPT-3.5 (2022)4k
GPT-4 (2023)8k / 32k
Claude / Gemini (2024)200k / 1M
LLaMA 3 (2024)128k
Lost-in-the-middle. Empirically, models recall information at the start and end of a long context far more reliably than information buried in the middle. A 1M-token window is not 1M tokens of working memory.

Sampling — how the next token is picked from the output distribution

$$p'_i = \frac{e^{z_i / T}}{\sum_j e^{z_j / T}}$$
Temperature T

T→0: deterministic (greedy). T=1: unmodified. T>1: flat distribution, more random, less coherent.

Use: 0.0–0.2 for code and retrieval, 0.7–1.0 for creative work.

Top-p (nucleus)

Keep the smallest set of tokens whose cumulative probability ≥ p. Adaptive: narrow when the model is confident, wide when it isn't.

Use: combine with T. Typical: T=0.7, top_p=0.9.

Top-k

Keep the k highest tokens and renormalise. Non-adaptive — cuts off high-probability tails in confident contexts and allows garbage in uncertain ones.

Use: rarely preferred over top-p.

🔒 The first dial: temperature. It does not add knowledge; it rescales what the model already believes. Low T hides the model's uncertainty by committing to the top guess. High T surfaces it and produces more variety — at the cost of correctness.
PART I · CH.1 LLM FUNDAMENTALS

1B — Pretraining → SFT → RLHF/DPO

Three stages turn a raw next-token predictor into the helpful, aligned assistant you call via API. Skipping or mixing up these stages is the most common conceptual error in interviews.

Stage 1 — Pretraining (scale)

Transformer decoder trained with cross-entropy next-token loss on 10–100 trillion tokens scraped from the web, books, and code. No human labels — the text itself is the supervision.

Learns: grammar, syntax, world knowledge, reasoning patterns latent in the data, multilingual text, code idioms, basic arithmetic.

Does not learn: to be helpful, to follow instructions, to prefer honest over fluent.

A purely pretrained model answers "What is the capital of France?" by continuing the text — which might be "?" or "A: Paris" or another question. It is not yet useful.

Stage 2 — Supervised Fine-Tuning (SFT) (instruction-follow)

Fine-tune the pretrained model on a curated dataset of (instruction, response) pairs written by humans. Cross-entropy applied only on the response tokens.

Input:   "Summarise this document in three bullets: [doc]"
Target:  "• Point 1\n• Point 2\n• Point 3"
Loss:    CE on target tokens only  ← critical detail

A few thousand high-quality examples produce a dramatic jump in instruction-following.

Risk: the model learns what annotators wrote, not what is correct. Verbose-confident annotators → verbose-confident model.

Stage 3 — RLHF / DPO (Alignment)

RLHF
  1. Sample two completions for the same prompt.
  2. Annotator picks the preferred one.
  3. Train a reward model R(prompt, completion) on preference pairs.
  4. Fine-tune with PPO against R, with a KL penalty toward the SFT model.
DPO (Direct Preference Optimization)

Skip the reward model. Train directly on preference pairs with a loss that raises P(preferred) and lowers P(rejected).

Now standard in open-source work — simpler, more stable, no PPO hyperparameter tuning.

The sycophancy trap. RLHF optimises for human approval, which is not the same as human benefit. Models learn to agree with the user's framing even when it's wrong. This is why you can "convince" an aligned model to change a correct answer by pushing back.
🔒 The second dial: base vs instruct. Base models are raw next-token predictors. Instruct/chat models have had SFT + alignment applied. For any application, use instruct. Base only for continued pretraining or research.

Q&A

Q: Explain the difference between RLHF and DPO.
RLHF trains a separate reward model R(prompt, completion) from human preference pairs, then fine-tunes the SFT model with PPO to maximise R, with a KL penalty to stay close to the SFT distribution. DPO skips the reward model entirely: it derives a closed-form loss directly on preference pairs that raises P(preferred) and lowers P(rejected). DPO is simpler, more stable, has fewer hyperparameters, and has largely replaced RLHF in open-source alignment.

Q: Why does top-p usually beat top-k?
Top-k uses a fixed candidate count regardless of the shape of the distribution. When the model is confident, top-k still samples from k−1 unlikely alternatives; when it's uncertain, top-k cuts off the long tail too aggressively. Top-p adapts: it keeps exactly as many tokens as are needed to cover p of the probability mass, so the candidate set is narrow when the model is confident and wide when it isn't.

Q: An LLM reliably gets 3+5 right but fails at 237+489. Why?
Numbers in BPE vocabularies typically split into per-digit or per-two-digit tokens. The model must learn arithmetic as a sequence prediction task over digit tokens, not as a numeric operation. Small numbers have enough coverage in training data to memorise; larger ones require generalising an algorithm the model was never directly taught. Chain-of-thought helps because it turns one hard step into many easy ones.

Q: The context window is 1M tokens. Why shouldn't I just stuff everything in?
Three reasons. (1) Cost: input tokens are billed. (2) Latency: prefill time grows roughly linearly with context length. (3) Lost-in-the-middle: attention reliably retrieves from the start and end of a long context but degrades in the middle — so burying a critical instruction at token 400,000 of 1,000,000 is a silent correctness bug. Prefer retrieval + smaller context.

12A — The Problem & the Handshake

Before MCP, every integration between an agent and a tool was a bespoke adapter. Team A writes a Python wrapper for the ERP. Team B writes a different wrapper for the same ERP. Team C writes yet another. With N agents and M tools, that's N×M bespoke integrations, each with its own error handling, auth, and logging quirks. MCP collapses it to N+M.

The × collapses to +

BeforeAfter
Agent A → ERP adapter3 MCP servers (ERP, Pricing, Email) written once — any number of agents discover and call them.
Agent A → Pricing adapter
Agent B → ERP adapter
Agent B → Pricing adapterN clients (one per agent framework) speak the shared protocol.

N×M custom adapters → N+M protocol endpoints.

The protocol: JSON-RPC 2.0 over stdio or HTTP+SSE

Lightweight RPC over JSON. Requires no HTTP (though supported). Request-ID matching lets a client multiplex concurrent calls over one connection.

// Client → Server: initialise
{"jsonrpc":"2.0","id":1,"method":"initialize",
 "params":{"protocolVersion":"2024-11-05",
           "capabilities":{"tools":{}}}}

// Server → Client: confirm + advertise
{"jsonrpc":"2.0","id":1,
 "result":{"protocolVersion":"2024-11-05",
           "capabilities":{"tools":{"listChanged":true}}}}

After the handshake the server is self-described. The client does not need prior knowledge of the server's capabilities — it discovers them at runtime.

Published November 2024 by Anthropic as an open standard. SDKs exist for Python, TypeScript, and Go. It is already the de-facto standard for tool integration across Claude Desktop, Cursor, and most agent frameworks.
🔒 The first lesson of MCP: discovery. Function calling hardcodes the tool schema into the client. MCP moves it server-side, where it can change without redeploying every agent.
PART II · CH.12 MCP

12B — The Three Primitives

MCP defines exactly three kinds of thing a server can expose. Telling them apart is the single most common interview test for this protocol.

1. Resources — read-only, URI-addressed data

The agent requests a URI; the server returns content. The agent does not mutate resources — that's what Tools are for.

@mcp_server.resource("order://{order_id}")
def get_order(order_id: str) -> str:
    return json.dumps(db.fetch_order(order_id))

# Client
content = await mcp_client.read_resource("order://PO-4812")

Typical uses: database records, file contents, API schema docs, product catalogues.

2. Tools — callable functions, may have side effects

The critical detail: tools/list returns the full JSON Schema of each tool's input. The agent never hardcodes or guesses the shape.

@mcp_server.tool()
def send_purchase_order(
        po_document: str,
        supplier_email: str) -> dict:
    result = email.send(
        to=supplier_email, body=po_document)
    return {"message_id": result.id,
            "status": "sent"}
{
  "name": "send_purchase_order",
  "inputSchema": {
    "type": "object",
    "properties": {
      "po_document": {"type":"string"},
      "supplier_email":
         {"type":"string","format":"email"}
    },
    "required": ["po_document",
                 "supplier_email"]
  }
}

3. Prompts — reusable, parameterised instruction templates

Domain-specific instruction engineering lives server-side, not in agent code. The client requests a prompt by name, passes parameters, receives a fully-formed message list.

@mcp_server.prompt()
def negotiate_price_prompt(supplier_name: str, target_price: float) -> list[dict]:
    return [
        {"role": "system",
         "content": f"You are a procurement specialist negotiating with {supplier_name}."},
        {"role": "user",
         "content": f"Target price: ${target_price:.2f}/unit. Begin negotiation."}
    ]
Why Prompts belong in MCP: when the business changes the negotiation stance, the agent binary does not need to be redeployed. The prompt template is a server-owned artifact.
🔒 The triage test: if the agent reads state → Resource. If it changes state → Tool. If it templates a prompt → Prompt. Same server can expose all three.

OrderFlow scenario — before and after

Before

Three teams, each owning one agent. Each maintained its own wrappers for ERP, Pricing, Email. Nine wrappers total. Each with subtly different error handling and retry behaviour. An ERP schema change was a three-team coordination problem.

After

Three MCP servers (one per system). Agents become MCP clients. Each team owns one server, zero agent-side adapters. ERP schema change = one deploy. Bonus: the compliance team dropped in a logging proxy MCP server in front of all tool calls — no agent code changed.

MCP does not replace RAG. MCP is the access/delivery layer; retrieval strategy, chunking, and reranking are still yours. The RAG pipeline can itself be packaged as an MCP Tool — that's composition, not replacement.

Q&A

Q: Walk me through deciding whether a capability should be a Resource or a Tool.
If the agent is reading state — a database record, a file, a catalogue entry — expose it as a Resource with a URI. Resources are idempotent reads. If the agent is changing state or causing side effects — sending email, booking a flight, writing to a queue — it's a Tool. When in doubt, ask "if I call this twice, does the world look different?" Yes → Tool. No → Resource.

Q: Why would you pick HTTP+SSE over stdio?
Three reasons. (1) Multiple concurrent clients: stdio pins the server to one parent process. (2) Independent scaling: an HTTP+SSE server lives behind a load balancer and scales on its own metrics. (3) Observability: HTTP middleware (auth, logging, rate-limiting, tracing) already exists at industrial strength. Use stdio for trusted, local, single-tenant tools; HTTP+SSE for production.

Q: A vendor exposes a "function-calling API." Is that the same as MCP?
No. Function calling is the model's ability to emit a structured tool-call token; MCP is a protocol for how tools advertise themselves and get invoked. The two compose: the model emits the function call; the agent framework looks up the tool over MCP and invokes it. MCP replaces hand-written adapters, not the model's tool-selection capability.

Q: How do you secure a multi-tenant MCP server?
Four layers. (1) Transport: TLS on HTTP+SSE, never plain stdio across a network. (2) AuthN: OAuth 2.0 bearer token; JWT claims identify the tenant. (3) AuthZ: server enforces per-tool ACLs — not every client can call every tool. (4) Input validation: even with a declared JSON Schema, validate again server-side; never trust the client's payload. See Ch.16 Trust & Sandboxing for the full defence stack.
